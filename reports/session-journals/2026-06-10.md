## 5. Insights and Learnings

### Technical insights

- **"Green CI, broken prod" had a single shape this session: real integration paths were never exercised.** The LLM path was mocked + cache-hit-only; the DB-write path was untested by the auth-only monitor; migrations weren't applied to prod at all. All three bit in production. The durable fixes were about *exercising the real thing*: a write-path smoke test, and auto-applying migrations against the real DB.
- **Supabase direct host is IPv6-only; GitHub Actions is IPv4-only.** Any CLI/migration step from Actions must use the **session pooler** (IPv4, port 5432, supports session-level DDL) — not `db.<ref>.supabase.co` and not the *transaction* pooler (6543, breaks migrations).
- **Re-running a failed GitHub Actions run replays that run's original commit**, not the current default branch. A workflow fix needs a fresh dispatch to take effect.
- **Adopting CLI migrations on a hand-managed DB requires a baseline.** With no `supabase_migrations` ledger, `db push` assumes nothing is applied and re-runs non-idempotent `CREATE TABLE`s. `migration repair --status applied` records existing migrations without executing them.
- **Features interact across caps.** INGEST-3 (80k parts) and the 16k comprehension cap composed into "comprehension never works on large docs" until COMP-6. Worth checking new outputs against neighboring limits.
- **Lossless splitting is a checkable invariant** (`"".join(parts) == text`), which made INGEST-3's chunker easy to trust. The standalone-fragment axe-scan technique was reused for the new part-nav.

### Process insights

- **Staging de-risks consequential infra.** Shipping the baseline as a *dormant, manually-triggered* workflow first (it can't affect deploys) let us discover and fix the pooler issue without ever touching the deploy path. Only after the ledger was verified did `db push` go into `deploy.yml`.
- **Verify flags/claims before shipping a "hardening."** Downloaded the pinned CLI to confirm `--yes` exists rather than guess — a wrong flag would have broken the very step it hardened.
- **The `ruff format --check` memory kept paying off** — caught locally on every Python PR; the only wasted CI cycle this session was an unrelated `supabase start` flake.

### Domain insights

- The product can now ingest and read **whole large documents** (auto-split into navigable parts) AND run comprehension on them (truncated input) — two gaps that previously forced manual splitting / showed "too long."
- Operationally, the schema-drift class of incident is now both **prevented** (OPS-3 auto-apply) and **detected** (OPS-4 write-path smoke), and the migration pipeline is proven on a real change (INGEST-3's columns landed via deploy).
