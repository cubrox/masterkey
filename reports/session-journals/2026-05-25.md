## 5. Insights and Learnings

### Technical insights

- **The dual-path shim pattern** (option D) is the right architectural choice for a "kill X but X has many consumers" migration. Steps:
  1. Add the new implementation alongside the old (don't delete)
  2. Make the entry point (here: `current_user`) check for the new path first, fall through to the legacy path on failure
  3. Migrate consumers one by one (tests, in our case)
  4. Once no consumers reference the legacy path, delete it atomically in a separate PR
  The shim's code is intentionally throwaway — the moment it's no longer needed, it's the destructive cleanup PR's job to rip it all out.

- **Auto-branching via Supabase GitHub App** is brittle in two ways:
  1. It needs to be *installed* on the repo separately from the project's "branching enabled" toggle (the Pro plan capability vs. the actual integration)
  2. It only triggers on `supabase/migrations/**` changes by default — config-only changes don't get a branch (correct, but surprising)

- **Cloud Run `--set-env-vars` is replace-semantics**: every deploy that uses this flag must enumerate the full env-var set. Dropping a value from the list removes it from the next revision. (Production-only `deploy-cloudrun@v2` uses the same semantics via its `env_vars:` block.) This means the closeout PR's "asserting the smaller list" implicitly deprovisions old vars without needing a separate `--remove-env-vars` step.

- **SQLModel `foreign_key="auth.users.id"` doesn't work** for FKs that cross schemas. The Supabase `auth` schema is not SQLModel-managed, so the FK declaration is meaningless at the ORM layer. Real enforcement lives in the SQL migration. SQLModel fields keep only the column type (`uuid.UUID`); the migration file declares the actual `REFERENCES auth.users(id)`.

- **`anon_client()` for auth verification, `service_client()` for cross-tenant admin work**, never crossed. The service key bypasses RLS entirely; one accidental import of the service client in a request handler defeats the entire security model.

### Process insights

- **Pre-emptive "additive vs. destructive" decomposition** when filing migration tickets prevents the mid-flight scope refactors we hit on SUPA-2 and SUPA-3. Lessons saved to memory but worth canonicalizing as a `/create-ticket` heuristic.

- **The conflict-of-interest disclosure pattern** on `/review-pr` (same session authored + reviewed) is correctly surfaced but doesn't actually prevent the agent from missing things. Eight required changes slipped through on #100's first review because the same model wrote the bug. A meaningful improvement here would be **a second reviewer agent** with strict instructions to look for the things the author *wouldn't* have written.

- **Memory-based feedback worked**: I caught the "GO with Required Changes" inconsistency *because* I'd saved a memory entry about it earlier in the session. The va-reviewer/va-worker switching gotcha got saved similarly. Both lessons are reusable across future sessions.

- **`/eli5` as a meta-tool**: dogfooding the new slash command against `#83` and `#96` during the same session confirmed the design. The biggest payoff was the Mermaid before/after diagram on #96 (the dual-path shim PR) — it made the architectural change immediate in a way the technical body couldn't.

### Domain insights

- The framework template's `agile-flow-meta` heritage shows in several places: stale Project Board URL in CLAUDE.md (`users/cubrox/projects/1` — cubrox is a repo, not a user), a `.claude/commands/doctor.md` that references a non-existent Neon CLI step, and the leftover session journals from a workshop-prep session. Worth auditing as a one-off "fork cleanup" pass.

- The Supabase Pro plan's branching cost model ($0.32/day per active branch) doesn't make per-PR isolation expensive for a typical cadence, but #99's per-PR-DATABASE_URL injection is the right defensive move once real users exist.

### 9.3 The auto-fix workflow defect

**`.github/workflows/auto-fix.yml` should not exist** in the upstream framework template as currently shaped. The `GITHUB_TOKEN`-signed auto-push silently breaks every PR with a fixable lint issue. Options for upstream:
1. Delete it (cubrox's choice — CI lint step in `ci.yml` already catches issues; pre-push hook covers local)
2. Switch to a PAT-based push (needs framework opinion on secret provisioning)
3. Switch to `--check`-only mode (fails the PR loud instead of auto-fixing)

### 9.4 The dual-path / shim refactor pattern

The "option D" migration pattern (keep the legacy SQLModel as a shim with a lazy-create layer, rip it out atomically in a follow-up PR) is reusable for any framework fork doing a major persistence-layer swap. Could be a new `docs/PATTERNS-FOR-MIGRATIONS.md` or appended to PATTERN-LIBRARY.

### 9.5 Multi-bot account scope hygiene

The default `va-reviewer` PAT scope set is `gist, read:org, repo, workflow` — missing `project`. This breaks the natural workflow of "review PR, then file a follow-up ticket on the project board" without an explicit account switch. Two options for upstream:
1. Default both bots to the same scope set including `project`
2. Document in `setup-accounts.sh` that reviewer + worker need different scopes and why

### 9.6 Session-journal cleanliness

The repo's initial commit (when forked from the framework template) included two crufty session journals (`2026-03-19.md`, `2026-04-17.md`) from `agile-flow-meta` work — not the user's project. Deleted in this PR. Worth either:
1. Removing example journals from the template's initial commit, or
2. Adding a step to `bootstrap-product` / similar that clears the journals directory on fork.

### 9.7 The `/work-ticket` pre-flight MCP check

The skill's pre-flight requires a GitHub MCP server (`Attempt a GitHub MCP tool call (e.g., list repos). If the MCP server is not connected, STOP.`). cubrox doesn't use a GitHub MCP — `gh` CLI is the documented path, per CLAUDE.md. This contradiction is benign for now but worth either:
1. Softening the pre-flight to "MCP **or** `gh` CLI"
2. Mandating MCP in the framework default

### 9.8 The "Required Changes = NO-GO" template enforcement

The `/review-pr` template has the words "Required Changes (if NO-GO)" but a human can easily ignore that and write "GO with one Required Change" (I did, once). The template could be strengthened with: "If any item appears under Required Changes, the recommendation MUST be NO-GO. No middle ground." (Saved as a session memory in this fork; framework-worthy.)
