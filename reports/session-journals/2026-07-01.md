## 5. Insights and Learnings

### Technical insights

- **Cloud Run traffic splits mean "latest revision" ≠ "serving revision".** `gcloud run services describe <svc> --format=yaml` shows the LATEST revision's config, not the one serving 100% traffic. Preview revisions with 0% traffic look identical in that output. Use `gcloud run revisions describe <specific-revision>` for the serving one. This bit us during #205 R13 verification — first dump showed preview-branch DB URL from smoke PR #241, only the revision-specific check confirmed prod actually had literal `DATABASE_URL`.
- **WIF `principalSet` binding is per-role, not global.** Both `workloadIdentityUser` AND `serviceAccountTokenCreator` must be bound for auth to complete — the token-creator role is what actually issues the impersonated credentials. Missing token-creator gets you past auth but denies `getAccessToken`. Our first preview-deploy failure was exactly this.
- **GitHub returns 404 for repo mutations you lack permission for**, not 403. Learned twice this session (rename + ruleset PUT). If gh returns 404 on `PATCH /repos/...` or `PUT /repos/.../rulesets/...`, first check `gh api repos/OWNER/REPO --jq .permissions` before assuming the resource doesn't exist.
- **Cloud Run env-var types are permanently sticky.** Once `DATABASE_URL` is created as `--set-env-vars` (literal), a subsequent `--set-secrets DATABASE_URL=...` bricks the service with "already set with a different type." R13 was our name for this; verified prod matches deploy.yml's expected shape.
- **`--tag=pr-<N>` for Cloud Run preview URLs is a stable-URL feature, not a separate service.** Preview deploys land as revisions on the SAME service with an alias tag; the URL is `https://pr-<N>---<service>-<hash>.run.app`. Cleaner than the "N services" alternative and matches how deploy.yml uses it.
- **Supabase's `gf_env` shim reads env vars in fallback order.** `GEMBAFLOW_*` preferred, `AGILE_FLOW_*` deprecated fallback. Prose docs should reference the preferred name; the fallback exists for back-compat during migration windows only.
- **Git repo rename cascades:** 30-day URL redirect covers external links; branch protection rulesets auto-follow (verified 15886599 still applied on cubrox/masterkey with same id); Actions secrets + variables stay; local `git remote` needs manual `set-url` per clone; WIF binding does NOT auto-follow (that's why dual-write via #200 was load-bearing).

### Process insights

- **The 4-phase multi-agent flow (architect → devops → QE → groomer) is worth its overhead for high-blast-radius refactors.** v3 caught 7 blockers in DevOps round 1; v4.1 caught 3 more in round 2. Skipping any pass would have shipped a broken plan.
- **Locking scope BEFORE dispatching agents saves enormous time.** All 3 "Decision" AskUserQuestions early in the session (repo rename? infra rename scope? project board?) let me brief architects with locked assumptions instead of them relitigating them.
- **When plans collapse execution steps, walk the ticket enumeration.** v4.1 §6 reconciliation of 53 tickets (8 CLOSE + 14 REWORK + 20 KEEP + 11 already-done) caught tickets whose bodies had drift. Without this, #201 would have failed pre-flight against inverted-reality checks.
- **Docs-only PRs still trigger `deploy.yml` on main** in this repo. Not path-filtered. Merges harmlessly re-deploy the same image. Not worth path-filtering unless deploys have side effects.
- **Bundling small tickets into a single PR is faster than 8 PRs when files are disjoint.** E2 (8 tickets, 8 commits, 1 PR #240) took ~20 min end-to-end. Groomer's insight was to match Epic #182's original design.
- **SOP: post explicit PR-link comments on tickets after `gh pr create`.** GitHub's auto-cross-reference is easy to miss in a busy comment stream. User flagged this mid-session; captured to memory as `feedback_pr-link-comment-sop.md`.
